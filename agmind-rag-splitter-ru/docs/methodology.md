# Методология и решения

Сквозные заметки по сборке русского context-aware RAG-сплиттера — выбор базы, PEFT, данные, деплой — с обоснованием каждого решения.

## 1. Базовая модель — `t-tech/T-lite-it-2.1`
Требования, определившие выбор (пересечение, не один фактор):
1. **Работает на llama.cpp Vulkan (AMD, без CUDA)** — это дисквалифицировало самые *новые* варианты: Qwen3.5-9B и Granite-4 — SSM/linear-attention гибриды без Vulkan-ядер (мусор на целевом железе). Новее ≠ лучше, когда архитектура ломает рантайм.
2. **Кириллически-эффективный токенайзер** — T-lite переработал словарь. Замер на RU-сэмпле (42 слова, `add_special_tokens=False`): T-lite **1.74 ток/слово**, Qwen2.5 ваниль 2.71, Llama-2 (датский оригинал) **3.17** — то есть T-lite почти вдвое компактнее Llama-2 на русском (×1.82). Подробный замер — `eval/results.md`.
3. **Apache-2.0** (commercial-OK) — отсёк YandexGPT-5 (кастомная NC) и Mistral MRL.
4. **Официальный GGUF существует** → round-trip кастомного токенайзера в GGUF уже доказан вендором.

Среди dense + Vulkan-safe + кириллица + Apache моделей T-lite-it-2.1 — самая свежая. Fallback: `RefalMachine/RuadaptQwen3-8B-Hybrid` (чуть лучше кириллический fertility, GGUF round-trip требует проверки).

## 2. PEFT — bf16 LoRA (r32) + rsLoRA, response-only
- **bf16, не QLoRA:** 8B влезает в 32 ГБ bf16 → 4-bit не даёт ничего, а bitsandbytes-4bit на Blackwell sm_120 хрупок + штраф по throughput (~58% от FP16). QLoRA — только fallback при OOM.
- **rsLoRA вкл.:** строго-лучший скейлинг (`alpha/√r`); бесплатно, разблокирует безопасный r=64 при необходимости.
- **Не DoRA / full-FT:** DoRA +13–49% времени и памяти (выталкивает 8B из bf16) без выигрыша на узкой задаче; full-FT ≈ LoRA на малых instruction-сетах («LoRA Without Regret»), но добавляет риск catastrophic-forgetting.
- **Response-only loss:** вывод (boundary JSON) крошечный против входа; без маскировки ~95% лосса ушло бы на воспроизведение входа.

## 3. Данные — дистилляция учителя с индексным выводом
- **Вывод индексов, а не переписывание текста.** Референсная датская модель выдаёт полный текст каждого чанка; это рискует байт-рассинхроном (модель «исправляет» ё/кавычки → чанки ≠ исходник → ломает цитирование в RAG) и ~в 10× медленнее. Вывод **индексов** + хост-сайд нарезка — lossless, быстро, и таблицы цельны по построению.
- **Учитель:** сильная модель размечает границы; self-consistency + жёсткие гейты заменяют ручную разметку.
- **Микс:** реальная проза (web/edu/тех) + синтетика таблиц/кода (12 доменов) для гарантии покрытия атомарности таблиц/кода. Динамический guard пропорций не даёт синтетике доминировать.

## 4. Деплой — GGUF на AMD Vulkan
- Конвертация merged FP16 → GGUF → квантизация Q5_K_M (~5.9 ГБ).
- **Граблина токенайзера:** хэш BPE-pre-tokenizer'а T-lite не зарегистрирован в upstream llama.cpp → `convert_hf_to_gguf.py` падает. Фикс: зарегистрировать хэш как `qwen2` (`training/patch_tokenizer_hash.py`).
- Проверка GGUF на целевом железе: boundary-F1 Q5/Vulkan-модели совпал с FP16 в пределах шума квантизации (0.639 vs 0.656 @0) — round-trip токенайзера + квантизации чистый.

## 5. Для чего это НЕ предназначено
Боундари-сплиттер обрабатывает прозу + мелкие/средние таблицы. **Огромные таблицы** (за бюджетом эмбеддера) требуют другой стратегии — построчная нарезка с повтором шапки + саммари таблиц + whole-table retrieval (parent-document по `table_id`) — это работа парсера документов + архитектуры ретрива, а не LLM-боундари-модели.

## Грабли (набитые на практике)
- Пинить Blackwell-зависимости точно (torch cu129 не cu130; выровнять torchaudio под torch; поднять `ulimit -n`).
- Использовать gamer-owned HF-кэш (root-owned `~/.cache/huggingface` молча падает как `PermissionError`).
- Выбирать чекпоинт по **task-метрике**, не по eval_loss.
- Проверять развёрнутый GGUF на реальном железе, не только на тренировочном.
